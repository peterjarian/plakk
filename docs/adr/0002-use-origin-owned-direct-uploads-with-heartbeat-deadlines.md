# Use origin-owned direct uploads with heartbeat deadlines

The origin client exclusively owns unfinished upload content and transfers it directly to the user's linked storage provider, keeping private bytes and gigabytes of data flow out of Plakk's backend. It requests a scoped upload destination from the backend, keeps at most one active transfer for that snippet, and reports success or failure afterward. A retry may request a replacement destination, but it remains the same snippet. The client-generated snippet identifier is also the create idempotency key; there is no separate client mutation identifier. The backend coordinates status rather than identifying or arbitrating competing workers. Origin ownership is determined by possession of the local outbox and managed bytes, not by a server credential. There is no upload lease identity, per-attempt identity, `BUSY` state, or preparation generation.

Because the backend cannot observe the direct transfer, the client extends an upload heartbeat deadline while it owns processing. The origin may retry a bounded number of times while the authoritative status remains `UPLOADING`. When those retries are exhausted, or when the heartbeat expires, the backend publishes non-terminal `FAILED`. Once shared, `FAILED` remains stable until the user explicitly retries on the origin or deletes the snippet; an explicit retry returns it to `UPLOADING`. Failure details and retry information remain local to the origin outbox, while other clients synchronize only `FAILED`. The backend is serverless, so #47 keeps the expiration operation callable from a short-lived invocation and tested while external cron wiring is deliberately deferred. These authoritative states are architecture and synchronization facts that clients translate into appropriate user-interface copy.

Authoritative metadata and upload status become visible to every client before file content is available. Clients show the snippet immediately and communicate whether it is uploading or failed rather than hiding it until completion. Every client sees `FAILED`; only a client with the snippet in its local outbox exposes upload recovery actions.

Deleting a snippet removes it immediately regardless of its upload status. The origin aborts unfinished work and removes managed local content, and the server publishes deletion when authoritative metadata exists. Cleanup of abandoned upload sessions or unknown partial objects is best-effort, while deletion of a known uploaded provider object creates durable, non-blocking cleanup work that retries until removal. Deletion always wins over a late completion; there is no separate cancellation status.

Closing the desktop window leaves the origin process and outbox running in the background. Explicit Quit is respected without blocking or warning: durable queued work remains on disk, heartbeats stop, and expiry may publish `FAILED`. On the next launch the origin presents the explicit retry action rather than automatically reversing that shared failure.
